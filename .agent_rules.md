# Протокол Разработки Antigrav (Antigrav Development Protocol)

Данный файл является обязательным к прочтению для ИИ-ассистента перед началом работы. Эти правила — закон проекта.

---

## 1. Архитектура и Модульность
- **Shared Components (Общие компоненты):** Все изменения в Шапке (Header), меню (Sidebar) и подвале (Footer) вносятся **ТОЛЬКО** в файл `components.js`. Прямое редактирование HTML-блоков в индивидуальных страницах запрещено.
- **Внедрение скриптов:** При создании новой страницы `.html` обязательно подключать `components.js` перед основным `script.js`.

## 2. Логирование и Память
- **DEVELOPMENT_LOG.md:** Каждая решенная проблема или ошибка (особенно критическая) должна быть зафиксирована в [DEVELOPMENT_LOG.md](file:///d:/Antigrav/DEVELOPMENT_LOG.md). 
- **Формат записи:** [Дата] - Описание проблемы - Причина (Анализ) - Решение - Лучшая практика на будущее.

## 2.5 Дизайн и Эстетика (Aesthetic Guide)
- **Aesthetic Alignment:** При создании новых UI-элементов сверяться с [AESTHETIC_GUIDE.md](file:///C:/Users/levda/.gemini/antigravity/brain/89f2f03b-0f2c-4fb7-9e52-41ed6a334b5f/AESTHETIC_GUIDE.md). Цвета, шрифты и анимации должны быть консистентны.
- **Интерактивность:** Все фото-образы должны быть динамическими и использовать данные из `looksData` в `products-data.js`.

## 3. Стабильность Кода (JavaScript)
- **Защитные проверки (Defensive checks):** Каждое обращение к DOM-элементу (`getElementById`, `querySelector`) и назначение слушателя (`addEventListener`) должно быть обернуто в проверку на существование элемента.
- **Обработка ошибок:** Критические блоки логики оборачивать в `try...catch` с выводом информативной ошибки в консоль (без падения всего сайта).
- **Санитизация:** Любой вывод текста из `products-data.js` в HTML должен проходить через функцию `sanitize()`.

## 4. Дизайн и Эстетика
- **Wow-эффект:** Любое визуальное изменение должно соответствовать премиальному стилю бренда (Burdundy, Gold, Cream). Использовать плавные переходы (`transition`) и `backdrop-filter` где это уместно.
- **Адаптивность:** Каждое UI-изменение должно проверяться на мобильных устройствах через браузерный инструмент.

## 5. Процесс Работы
- **Git Commits:** По завершении задачи генерировать краткий и понятный текст для коммита на русском языке по стандарту: `<тип>: <краткое описание>` (например, `feat:`, `fix:`, `refactor:`, `style:`).
- **Валидация ресурсов:** Перед сдачей работы по наполнению каталога запускать `node check-assets.js` (если доступен Node) или проверять пути вручную.
- **Верификация:** По запросу пользователя проводить автоматизированный тест через браузер на проверку работоспособности меню и основных кнопок.
